# macOS security-scoped bookmarks: reliability and recovery

Research date: 2026-07-18

Scope: primary Apple sources only — current Apple documentation, Foundation API documentation, and Apple Developer Forums answers from Apple DTS engineers.

## Bottom line

Use a security-scoped bookmark, not a stored path, when a sandboxed macOS app must retain a user's file or folder grant across launches. A bookmark does **not** copy the selected file or folder contents: Apple defines it as an opaque data structure that describes a file's location, and resolving it produces a URL through which the app accesses the original live resource ([File System Programming Guide](https://developer.apple.com/library/archive/documentation/FileManagement/Conceptual/FileSystemProgrammingGuide/AccessingFilesandDirectories/AccessingFilesandDirectories.html), [`URL` bookmark-data API](https://developer.apple.com/documentation/foundation/nsurl/bookmarkdata%28options%3Aincludingresourcevaluesforkeys%3Arelativeto%3A%29)). Security scope adds the persisted user grant to that reference; it is not a snapshot or backup. Apple describes bookmarks as persistent references that usually survive moves, renames, app relaunches, and system restarts, subject to the capabilities of the underlying volume ([`NSURL` overview](https://developer.apple.com/documentation/foundation/nsurl), [`bookmarkData` API](https://developer.apple.com/documentation/foundation/nsurl/bookmarkdata%28options%3Aincludingresourcevaluesforkeys%3Arelativeto%3A%29)).

Bookmarks are durable, not infallible. The application should implement three distinct outcomes:

1. **Resolved and current:** start scoped access, use the resource, and stop access.
2. **Resolved but stale:** immediately create a replacement bookmark from the resolved URL and replace the stored data before continuing.
3. **Resolution or access failed:** retain the error, use the bookmark's last-known metadata to explain or seed recovery, and ask the user to select the resource again when access cannot be restored.

That flow follows Apple's current [macOS App Sandbox file-access guide](https://developer.apple.com/documentation/security/accessing-files-from-the-macos-app-sandbox) and the [`URLByResolvingBookmarkData` contract](https://developer.apple.com/documentation/foundation/nsurl/urlbyresolvingbookmarkdata%3Aoptions%3Arelativetourl%3Abookmarkdataisstale%3Aerror%3A?language=objc).

## Paths, symlinks, aliases, and bookmarks

These mechanisms solve different problems. None copies the target's contents; copying into app-controlled storage is a separate operation.

| Mechanism | What is persisted | If the target moves or is renamed | Does it retain App Sandbox access across launch? | Fit for this app |
| --- | --- | --- | --- | --- |
| Plain stored path or file URL | A pathname identifying a location. Apple notes that path-based URLs become invalid when the referenced item moves ([File System Programming Guide](https://developer.apple.com/library/archive/documentation/FileManagement/Conceptual/FileSystemProgrammingGuide/AccessingFilesandDirectories/AccessingFilesandDirectories.html)). | No. It may instead refer to a different item later created at the same path. | No. Apple says the sandbox extension created by an open panel applies to the selected URL, while a security-scoped bookmark is the mechanism for access after exit and relaunch ([App Sandbox file-access guide](https://developer.apple.com/documentation/security/accessing-files-from-the-macos-app-sandbox)). | Keep a path only as display and last-known-location metadata, never as the durable identity or grant. |
| POSIX symbolic link | A separate lightweight file containing an absolute or relative pathname to its target ([Apple HFS Plus specification](https://developer.apple.com/library/archive/technotes/tn/tn1150.html#SymbolicLinks)). | Generally no: because resolution is path-only, moving or renaming the target breaks the link; Apple contrasts this with aliases and says repair requires recreating the symlink ([Aliases and Symbolic Links](https://developer.apple.com/library/archive/documentation/MacOSX/Conceptual/BPFileSystem/Articles/Aliases.html)). A relative link can remain valid only when the relative relationship still resolves. | No. Apple explicitly says a symlink inside an app container does not make an outside target accessible without the appropriate permission ([sandbox-container migration guide](https://developer.apple.com/documentation/security/migrating-your-app-s-files-to-its-app-sandbox-container)). | Useful for user-controlled filesystem topology, not for recording consent or making a workspace root durable. |
| Finder alias file | A lightweight reference backed by bookmark data; the `suitableForBookmarkFile` option adds the properties required for a Finder alias ([creation option](https://developer.apple.com/documentation/corefoundation/cfurlbookmarkcreationoptions/suitableforbookmarkfile)). | More resilient than a symlink: Apple documents pathname plus persistent identity lookup and same-volume move recovery, while still treating resolution as fallible ([Aliases and Symbolic Links](https://developer.apple.com/library/archive/documentation/MacOSX/Conceptual/BPFileSystem/Articles/Aliases.html)). | Not the app's persistent grant mechanism. Foundation explicitly forbids combining `withSecurityScope` with `suitableForBookmarkFile` ([`withSecurityScope`](https://developer.apple.com/documentation/foundation/nsurl/bookmarkcreationoptions/withsecurityscope?language=objc)). | Appropriate as a Finder-visible shortcut if the product ever needs one; do not use it as internal access state. |
| App-scoped security-scoped bookmark data | Opaque bookmark/reference data plus explicit security scope. Resolution yields a security-scoped URL; only then can the app activate access ([App Sandbox file-access guide](https://developer.apple.com/documentation/security/accessing-files-from-the-macos-app-sandbox), [`startAccessingSecurityScopedResource`](https://developer.apple.com/documentation/foundation/url/startaccessingsecurityscopedresource%28%29)). | Usually yes for moves and renames when the volume format supports it, with explicit stale and failure outcomes ([`bookmarkData` API](https://developer.apple.com/documentation/foundation/nsurl/bookmarkdata%28options%3Aincludingresourcevaluesforkeys%3Arelativeto%3A%29), [resolver API](https://developer.apple.com/documentation/foundation/nsurl/urlbyresolvingbookmarkdata%3Aoptions%3Arelativetourl%3Abookmarkdataisstale%3Aerror%3A?language=objc)). | Yes, for the creating app's signing identity, provided the required entitlements are present and access is successfully started ([`bookmarkData` API](https://developer.apple.com/documentation/foundation/nsurl/bookmarkdata%28options%3Aincludingresourcevaluesforkeys%3Arelativeto%3A%29), [security-scope entitlement guide](https://developer.apple.com/documentation/professional-video-applications/enabling-security-scoped-bookmark-and-url-access)). | The correct representation for a linked external Primary Folder. Use read-only scope when source modification is not required. |

## Recommendation for the local-first app

For each optional external Primary Folder, have the user choose the directory through `NSOpenPanel`, immediately create an app-scoped security-scoped bookmark, and persist the opaque bookmark in app-controlled metadata. Because the product does not need to modify source material, request read-only scope; Apple provides `securityScopeAllowOnlyReadAccess` for that purpose ([App Sandbox file-access guide](https://developer.apple.com/documentation/security/accessing-files-from-the-macos-app-sandbox), [read-only creation option](https://developer.apple.com/documentation/foundation/nsurl/bookmarkcreationoptions/securityscopeallowonlyreadaccess)). Store a display name and last-known path separately for UI and recovery, but never treat either as authority.

Do not create a symlink or Finder alias as a permission workaround. They may be useful user-facing filesystem objects, but neither supplies the app-scoped, relaunch-persistent access contract. If a Managed Asset must remain available independently of its source, explicitly copy that asset into the app container; a security-scoped bookmark deliberately keeps the linked resource external and live.

## Reference options compared

| Stored form | Move or rename behavior | Persistent sandbox permission | Fit for this app |
| --- | --- | --- | --- |
| Plain path or path-based URL | Names the route through the directory hierarchy, so moving or renaming the target invalidates it. | None. A path is only an address. | Keep only as user-facing, last-known metadata and a picker hint. |
| POSIX symbolic link | Stores a path to its target. Apple says a same-volume move breaks symbolic links and the link must be recreated. | None. Apple explicitly says a sandboxed app cannot access an outside target merely because a symlink to it exists inside the app container. | Useful only where another system requires normal filesystem link semantics; it does not solve either durable permission or general move tracking. |
| Finder alias file | A lightweight location reference rather than a copy. It combines path and file identity information and can survive a same-volume move better than a symlink. Apple's bookmark API can write bookmark data to disk as an alias file. | Not the API Apple specifies for retaining an App Sandbox grant across launches. Resolving a location and receiving sandbox authority remain separate requirements. | Relevant for Finder-visible interoperability, but unnecessary internal indirection for app persistence. |
| App-scoped security-scoped bookmark data | Usually resolves after a move or rename, subject to filesystem support; explicitly reports staleness for refresh. | Yes. This is Apple's supported mechanism for preserving user-approved access across launches. | **Recommended canonical record**, stored in the app's database or application-support state. |

Apple's path and bookmark distinctions come from the current [File System Programming Guide](https://developer.apple.com/library/archive/documentation/FileManagement/Conceptual/FileSystemProgrammingGuide/AccessingFilesandDirectories/AccessingFilesandDirectories.html). The symlink-versus-alias behavior is stated directly in Apple's retired but still first-party [Aliases and Symbolic Links](https://developer.apple.com/library/archive/documentation/MacOSX/Conceptual/BPFileSystem/Articles/Aliases.html) guide; the current sandbox documentation separately confirms that [a symlink does not bypass access restrictions](https://developer.apple.com/documentation/security/migrating-your-app-s-files-to-its-app-sandbox-container).

The key distinction is that **location indirection is not authorization**. A symlink may redirect an already-authorized file operation, but it cannot mint or preserve a sandbox extension for its target. A Finder alias improves location recovery, but Apple directs sandboxed apps that need persistent external access to security-scoped bookmarks. The app should therefore store one security-scoped bookmark as the canonical capability, plus a plain last-known path for display and recovery; it should not create symlinks or Finder alias files as an internal permission mechanism.

## Recommended lifecycle

- Create bookmark data from the system-provided URL with `.withSecurityScope`. For read-only future access, also request the read-only security-scope option ([Apple file-access guide](https://developer.apple.com/documentation/security/accessing-files-from-the-macos-app-sandbox)).
- Resolve stored data with `.withSecurityScope`.
- If `bookmarkDataIsStale` is `true`, create new bookmark data from the returned URL and replace all stored copies. Staleness is recoverable because resolution still returned a URL ([resolver API](https://developer.apple.com/documentation/foundation/nsurl/urlbyresolvingbookmarkdata%3Aoptions%3Arelativetourl%3Abookmarkdataisstale%3Aerror%3A?language=objc)).
- Call `startAccessingSecurityScopedResource()` before using a URL resolved from an explicitly security-scoped bookmark. If it returns `true`, balance it with `stopAccessingSecurityScopedResource()` as soon as work finishes. Apple warns that unbalanced access leaks kernel resources and can prevent the app from adding further sandbox locations until relaunch ([`NSURL` overview](https://developer.apple.com/documentation/foundation/nsurl)).
- Treat a `false` result from `startAccessingSecurityScopedResource()` as a failed access grant. Apple DTS says that on macOS this generally means the app does not have access and the flow or entitlements need investigation ([DTS thread 798402](https://developer.apple.com/forums/thread/798402?answerId=855880022)).
- Keep the bookmark data and the resolved URL under one owner that manages access lifetime. Apple DTS recommends immediately converting a system-provided URL into a bookmark, resolving it, discarding the original URL, and making the rest of the app use the bookmark-derived URL consistently ([DTS thread 798402](https://developer.apple.com/forums/thread/798402?answerId=855880022)).

## Failure boundaries and recovery

### Resource missing or volume unavailable

Apple documents that resolution can fail when the original file or directory cannot be located or its volume cannot be mounted. When it fails, read bookmark metadata with `resourceValues(forKeys:fromBookmarkData:)`; the stored `.pathKey` can provide the last-known path to help the user decide what to select next ([resolver API](https://developer.apple.com/documentation/foundation/nsurl/urlbyresolvingbookmarkdata%3Aoptions%3Arelativetourl%3Abookmarkdataisstale%3Aerror%3A?language=objc)). Because no URL was returned, this is not the stale-bookmark refresh case. Recovery should reacquire the user's grant through an open panel and replace the stored bookmark.

### Signing identity changed

An app-scoped security bookmark fails to resolve when the resolver does not have the same code-signing identity as the creator ([`bookmarkData` API](https://developer.apple.com/documentation/foundation/nsurl/bookmarkdata%28options%3Aincludingresourcevaluesforkeys%3Arelativeto%3A%29)). Keep the production identity stable. If an ownership or signing transition breaks existing grants, the safe recovery is to ask the user to select the resource again and store a newly created bookmark.

### Cross-process use

Apple's current guidance says explicit security-scoped bookmarks are necessary when access must survive app exit and relaunch. For passing access to another running process, such as an XPC service or launch agent, create a normal bookmark without explicit security scope; the receiving process gets the implicit scope and can defer activation with `.withoutImplicitStartAccessing` ([Apple file-access guide](https://developer.apple.com/documentation/security/accessing-files-from-the-macos-app-sandbox)). Do not assume an app-scoped security bookmark created by one executable is a portable capability for another executable.

### Filesystem and storage limitations

Apple DTS states that the intended contract is to bookmark everything returned by Apple's open panel and resolve it within the limits of the filesystem or storage provider. Resolution relies on metadata supplied by the filesystem, so third-party filesystems may not behave like Apple's filesystems. Duplicated disk images can also make volume identity ambiguous because two mounted sources can legitimately present the same identity ([DTS thread 798402](https://developer.apple.com/forums/thread/798402?answerId=855880022)).

### Operating-system regressions

- On macOS 15.0, Apple confirmed that `ScopedBookmarksAgent` could hang when launched while the keychain was locked, causing creation and resolution failures with `NSCocoaErrorDomain` code 256 and `Failed to retrieve app-scope key`. Apple said the bug was fixed in macOS 15.1. During development, kill the agent and possibly relaunch the app; for users, log out or reboot ([DTS thread 764435](https://developer.apple.com/forums/thread/764435)).
- Apple later confirmed macOS 26.0 bug `r.157722315`, which prevented security-scoped bookmark creation for the root directory of every filesystem. For a root-scanning application, Apple's temporary workaround was to bookmark each root child and require the user to select the root again if a later root listing differs ([DTS thread 799829](https://developer.apple.com/forums/thread/799829)).

Apple DTS notes that framework, daemon, sandbox, and filesystem failures can surface as the same top-level bookmark error. For a failure on a supposedly fixed OS, capture the precise OS build, filesystem or volume type, operation, and timing, then file a new Feedback rather than assuming it is the old bug ([DTS thread 764435](https://developer.apple.com/forums/thread/764435)).

## Product implication

Model persisted access as a recoverable permission, not an invariant. Store the opaque bookmark plus user-facing resource metadata; refresh stale data eagerly; make `start`/`stop` balancing structural; and provide a clear reauthorization path that opens the system picker near the last-known location. A path can help the user find the target, but it is not a substitute for the security-scoped bookmark that restores sandbox access.
