# OpenAI Build Week — Official Hackathon Reference

> **Source of truth:** [OpenAI Build Week on Devpost](https://openai.devpost.com) and its [Official Rules](https://openai.devpost.com/rules).
> **Status checked:** 14 July 2026. The rules control if this reference and the live site differ.

This is the working, comprehensive reference for the team. It deliberately retains all material requirements and operational detail; we can later make a shorter build checklist from it.

## At a glance

- **Host:** OpenAI; administered by Devpost.
- **Format:** online, global; submissions are open.
- **Challenge:** build a working project with **Codex and GPT-5.6** that fits exactly one track.
- **Submission deadline:** **Monday, 21 July 2026, 5:00 PM Pacific Time** (22 July, 12:00 AM UTC).
- **Submission window:** 13 July, 9:00 AM PT through 21 July, 5:00 PM PT.
- **Judging:** 22 July, 10:00 AM PT through 5 August, 5:00 PM PT.
- **Winners:** on or around 12 August 2026, 2:00 PM PT.
- **Prize pool:** $100,000 USD across eight awards.

## Tracks / what to build

Build an app, agent, website, game, workflow, developer tool, or another project type using Codex with GPT-5.6. Select the one category that best fits the project:

1. **Apps for Your Life** — consumer apps for everyday life: productivity, creativity, home, family, travel, health, or personal finance.
2. **Work & Productivity** — tools that make teams faster or more effective: workflow automation, customer support, analytics, sales, or back-office operations.
3. **Developer Tools** — developer-facing testing, DevOps, agentic workflows, or security tools.
4. **Education** — projects advancing AI for students, teachers, or educational organizations.

The project must install and run consistently on its intended platform, and function as shown in the video and/or stated in the description. Third-party SDKs, APIs, and data may be used only where their terms and licences permit it.

### New versus existing projects

- A project must be newly made during the submission period **or** meaningfully extended with Codex and/or GPT-5.6 after 13 July 2026, 9:00 AM PT.
- A pre-existing project is judged only on work added during the submission period.
- For a pre-existing project, document the boundary between prior and new work and retain evidence of period-appropriate Codex/GPT-5.6 use (for example, timestamped Codex session logs or dated commits).
- More than one submission is allowed only when each is unique and substantially different.

## Required submission package

Submit through the [Devpost submission page](https://openai.devpost.com) during the submission period. A saved draft is not a submission.

| Item | Requirement |
| --- | --- |
| Working project | Built with Codex and GPT-5.6; meets the project requirements above. |
| Category | Select one of the four tracks. |
| Project description | Explain the project’s features and functionality. Do not submit unedited AI-generated copy. |
| Demo video | **Public YouTube**, under **3 minutes**. It must clearly show the project working and include audio/voiceover explaining what was built, how Codex was used, and how GPT-5.6 was used. Judges need not watch beyond 3 minutes. No unlicensed music, third-party marks, or other material. A music-only screencast does not qualify. AI-assisted narration is allowed. |
| Code repository URL | Required. Public repos need an appropriate open-source licence. Private repos must be shared with **testing@devpost.com** and **build-week-event@openai.com**. |
| README | Required in the code repo. Include setup instructions, any sample data, and clear run instructions. Explain how the team collaborated with Codex, where Codex accelerated the workflow, key product/engineering/design decisions, and how Codex and GPT-5.6 contributed to the result. |
| `/feedback` Codex Session ID | Required: the session ID for the Project thread where the majority of core functionality was built. |
| Testing access | Provide a website, working demo, or test build. For a private site, give login credentials in the testing instructions. It must be freely available without restriction for sponsor, Devpost, and judges through the end of judging. |
| Plugin/dev-tool details | If applicable, include installation instructions, supported platforms, and a practical no-rebuild testing route (demo instance, sandbox, or test account). |

### Live form fields

The Devpost form currently requires:

1. Submitter type: Individual, Team of Individuals, or Organization.
2. Country of residence.
3. Category.
4. Code-repository URL (with README and Codex/GPT-5.6 usage highlighted).
5. `/feedback` Codex Session ID.

It optionally accepts a judge-access project link and testing instructions (including non-public credentials), plus plugin/dev-tool installation and testing instructions. The form says the judge-access instructions are visible only to Devpost managers and judges.

### Pre-submit checklist

- [ ] Project is working, runnable, and in exactly one track.
- [ ] Public YouTube demo is under three minutes and has a voiceover covering the build, Codex, and GPT-5.6.
- [ ] Repository is public with a suitable licence, or shared with both required judging addresses.
- [ ] README covers setup, data, running, and the substantive Codex/GPT-5.6 collaboration narrative.
- [ ] Judge testing route and credentials are present where needed and remain free through judging.
- [ ] `/feedback` Session ID is entered.
- [ ] All team members have accepted their Devpost invitations; an eligible representative submits for a team/organization.
- [ ] Submission is published, not left as a draft.

## Judging

Stage one is a pass/fail baseline: the idea must fit the theme and reasonably use the required technologies. Stage two scores all passing submissions **equally** on:

1. **Technological Implementation** — thorough, skillful Codex use; genuine effort and a working, non-trivial implementation.
2. **Design** — a runnable project with a complete, coherent product experience, not merely a technical proof of concept.
3. **Potential Impact** — a credible, specific real problem and audience, with a demonstrated solution.
4. **Quality of the Idea** — creativity and genuine understanding of the problem space.

Ties are resolved first by the earlier criterion in the order above, then later criteria, then a judges’ vote. Judges may use the description, images, and video alone; do not assume they will build the project.
