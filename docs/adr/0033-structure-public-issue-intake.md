# Structure public issue intake

Clarifold will accept public feedback through four GitHub issue forms: bug reports, learning-experience proposals, mathematical-accuracy concerns, and accessibility or usability issues. Each form must collect information appropriate to its subject, warn reporters not to include private learner material or secrets, and apply `needs-triage`; only maintainer-approved, fully specified implementation work may later receive `ready-for-agent`. Blank issues will be disabled. The issue chooser must direct suspected vulnerabilities to GitHub Private Vulnerability Reporting with `security@jeromegroup.org` as fallback, and conduct reports to `conduct@jeromegroup.org`. Public issue intake does not change the existing decision not to accept outside code contributions or treat pull requests as a request surface.

During beta, Clarifold provides no dedicated support service, response-time promise, general-support inbox, or GitHub Discussions forum. Reports are reviewed on a best-effort basis and submission does not guarantee a response, acceptance, or implementation. Setup troubleshooting belongs in the development guide. Public issue surfaces must state that they are not appropriate for private learner material, emergencies, security disclosures, or conduct reports. This boundary must be reconsidered when real usage and maintainer capacity justify a support channel.

GitHub enforces the protective-phase participation boundary with `pull_request_creation_policy: collaborators_only`: Issues and pull requests remain enabled for structured feedback, maintainer work, approved collaborators, and repository automation, while unsolicited outside pull requests are blocked. Discussions remain disabled, and the unused Wiki and repository Projects surfaces are disabled so public documentation and planning have one maintained home. These settings were applied and verified on `jerome-queck/clarifold` on 2026-07-21.

Clarifold will not add `CODEOWNERS` while Jerome Queck is the sole owner and maintainer, outside pull requests are blocked, and no independent approval is required. Adding the first trusted collaborator, assigning a specialist ownership area, or changing pull-request review requirements must trigger an explicit governance review of `CODEOWNERS`, required reviewers, bypass permissions, and the documentation of decision authority. The collaboration setup is incomplete until that review is resolved and recorded.
