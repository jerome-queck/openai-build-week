# Separate product, development, and architecture documentation

Clarifold's README will serve prospective users first while retaining a compact build-from-source gateway. Its status block will say: “Clarifold is an early, source-available macOS beta. The current source is available, and verified development builds are produced for technical evaluation. Clarifold does not yet provide a signed and notarized installer for ordinary macOS users. Jerome Queck’s original code is licensed for permitted noncommercial use under PolyForm Noncommercial 1.0.0. Commercial use requires prior written permission.” After the application rebrand, it will show three genuine packaged-app screenshots with safe fixture data and accessible alt text: starting advanced-mathematics work, exposing an understanding gap, and reviewing durable evidence, provenance, or formal verification. Mockups, stale Quick Study product branding, learner data, and an unmaintained animated walkthrough are excluded. Detailed setup, command, verification, packaging, and troubleshooting guidance will move to one canonical development guide, while stable component responsibilities and public engineering seams will move to one canonical architecture guide; other documents link to those owners instead of copying their detail. The repository engineering workflow, coding standards, review surfaces, and automated documentation checks must enforce this ownership without modifying the vendored Matt skills, so an implementation updates its affected code, executable configuration, and owning human document together. Every material pull request must declare whether it affects product guidance, development and verification guidance, architecture or standards, or contribution, security, licensing, and governance material; declaring no documentation impact requires a reason, and an incorrect or missing declaration is a Standards review finding. The normal verification lane must run a deterministic documentation check for local links and anchors, documented npm scripts, required canonical documents and policy sections, and the README's developer-guide routes; external-network availability and semantic prose accuracy remain outside that mechanical check.

The initial public-repository sweep will not add a separate `SUPPORT.md`, `GOVERNANCE.md`, `FUNDING.yml`, or `CITATION.cff`. Clarifold has no promised support service, is governed by its sole owner during the protective phase, is not soliciting funding through GitHub, and is not yet being published as a citable research artifact. The README, issue chooser, contribution policy, and existing authority documents will carry the relevant boundaries without creating duplicate surfaces that must be maintained. Each deferred file may be reconsidered when its underlying condition changes.

Clarifold will also provide an in-app **About Clarifold** surface. It must show the app version and build identifier, `Copyright © 2026 Jerome Queck`, the license and commercial-permission route, privacy policy, private security-reporting route, repository and issue tracker, the active local data location, and export and deletion guidance. Build identity and links must come from canonical application metadata or repository policy configuration rather than independently copied strings. The surface is a durable legal, identity, and data-control reference; it does not imply a general-support service or response-time commitment.

The README must include a concise **What Clarifold does—and does not claim** section. It will explain that Clarifold helps learners uncover and examine gaps in mathematical understanding but does not guarantee mastery, correctness, or academic outcomes; formal verification supports only the exact claims and assumptions actually checked; AI-generated teaching or analysis may be incomplete or wrong; and local-first operation does not mean every optional model or research operation remains offline. It must also preserve learner control over what is saved, sent, exported, and deleted. These statements must link to the canonical privacy, verification, and data-control explanations where more detail is needed rather than becoming an isolated disclaimer that can drift.

The README will use a one-badge policy: after the renamed verification workflow is stable, it may show only that workflow's current CI status. It will not use open-source, free, download-count, coverage-percentage, AI-generated, or decorative vanity badges. A badge must communicate a maintained fact without misrepresenting Clarifold's source-available license, distribution readiness, or product quality.

Clarifold will not add a public `ROADMAP.md` during this sweep. The README will instead provide a short project-status section describing the present beta stage and near-term priority areas without dates or delivery promises. GitHub Issues will remain the authority for concrete accepted work: only maintainer-approved issues may be read as planned, while proposals, reports, and exploratory ideas are not commitments. Even an accepted issue does not promise a release date.

Pull-request CI will use an always-present required gate that classifies the changed paths and reports which verification surfaces ran. Linting, typechecking, unit tests, documentation-policy checks, and the classification gate run for every material pull request. Packaging and packaged smoke may be skipped only when the classifier proves that the diff cannot affect runtime code, native helpers, bundled verifier assets, dependencies, packaging, or workflows; a documentation-only change still runs its owning documentation checks. Pushes to `main`, scheduled verification, manually dispatched candidates, and release candidates run the complete repository verification lane. The required aggregate check must fail closed when a path is unclassified so path filtering cannot silently bypass evidence.

The repository will use a conservative, collaborator-friendly ignore policy. Repository-level ignores are limited to unambiguously local or reproducible material: operating-system metadata; installed dependencies; generated build, package, test, coverage, cache, and log output; local environment files with example templates explicitly retained; and temporary editor swap or backup files. The repository will not broadly ignore `.vscode/`, `.idea/`, certificate or key file extensions, or general data formats, because those may contain deliberately shared configuration or legitimate fixtures. Private signing assets, when introduced, must live in a specifically named ignored private directory and remain protected by the signing workflow; `.gitignore` is not a security control. Reviewed golden fixtures belong in an intentional committed fixture directory rather than being force-added from a generated-output directory.

Adding the first collaborator must trigger an onboarding review of shared editor configuration, personal exclusions, committed fixtures, generated paths introduced by the active toolchain, environment-example coverage, and the private signing boundary. Personal editor exclusions should normally use a contributor's global Git ignore or `.git/info/exclude`; useful project configuration may be committed and reviewed. Collaboration onboarding is incomplete until this ignore-policy review is recorded alongside the `CODEOWNERS` and review-authority decision.

The README will include a short accessibility section that identifies accessibility as a product requirement and routes reports to the accessibility and usability issue form. It may describe only capabilities verified in the current build and must not claim WCAG conformance, full accessibility, or an equivalent status without a formal review. Before the first signed public installer, Clarifold must complete an accessibility review covering the packaged macOS experience. Any later conformance claim must identify the standard, version, level, platform and feature scope, audit date, testing basis, and known exceptions so that the claim remains falsifiable and maintainable.
