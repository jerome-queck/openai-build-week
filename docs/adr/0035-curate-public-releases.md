# Curate public releases

Clarifold will use GitHub Actions artifacts for internal candidate builds and GitHub Releases only for builds deliberately published to users. Pull-request CI may skip packaging only when an always-present, fail-closed classifier proves that the diff cannot affect the application artifact; every `main` integration, scheduled full verification, manually dispatched candidate, and release candidate still packages and verifies the application. Routine CI must not retain the approximately 1.1 GB application archive from every successful run. Small test reports and diagnostics are retained for 14 days. A manually dispatched release-candidate workflow retains the exact verified application archive for 30 days without creating a permanent Git tag. Publishing must promote that same verified binary rather than rebuild it. A public beta receives an immutable semantic prerelease tag such as `v0.2.0-beta.1`, a GitHub prerelease entry, the tested macOS artifact, checksum and provenance, supported-system information, known limitations, privacy and support links, and release notes. Replacement betas increment the prerelease number rather than publishing a permanent candidate tag for every commit. Published tags are never moved to another commit. GitHub Packages is not used for the macOS desktop binary; it may be reconsidered only if Clarifold later publishes a package-manager-native library, container, or other supported package format.

The Clarifold application-identity and data migration moves the working package line to `0.2.0`, but no public `v0.2.0-beta.1` release is promised. Until Developer ID signing and notarization are available, Clarifold publishes no prebuilt application through GitHub Releases; its status is a source-available development preview and temporary CI candidates are clearly marked not for distribution. When distribution becomes feasible, the first public beta receives the then-current appropriate semantic prerelease version rather than reviving a stale reserved tag. A non-prerelease version remains reserved for an intentional declaration that the line is stable; rebranding alone does not make it stable.

On 2026-07-21, after withdrawal from Build Week, the three `macos-beta-0.1.0-candidate-*` GitHub prereleases and their matching tags were deleted. Their assets had recorded zero downloads, they were not supported public distributions, and GitHub Releases and the remote candidate-tag namespace were verified empty afterward.

Clarifold will not maintain `CHANGELOG.md` before its first signed and notarized public beta. During the development-preview phase, Git history and merged pull requests record engineering changes. Each future public beta receives reviewed GitHub Release notes; when the first official public beta exists, introduce a changelog only if users need a durable upgrade history and make its generation or maintenance part of the release workflow. Historical Quick Study work is not rewritten as if it had shipped publicly under Clarifold.
