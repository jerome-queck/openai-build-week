# Defer public macOS binaries until notarized

Clarifold will not publish a prebuilt macOS application for ordinary learners until Jerome Queck is ready to enroll in the Apple Developer Program. During this phase the public repository presents an early source-available macOS beta: the current source remains available, verified development builds are produced for technical evaluation, the README leads with the product and genuine screenshots, and it states plainly that no signed and notarized installer is available for ordinary macOS users. It links developers to the canonical build-from-source guide and never instructs users to bypass Gatekeeper. Temporary CI archives are verification evidence named `internal-candidate-not-for-distribution`, not GitHub Releases or supported downloads.

After Apple Developer Program enrollment and before the first public binary, a dedicated release workstream must obtain and securely manage a Developer ID Application identity; enable and verify Hardened Runtime with the minimum justified entitlements; correctly sign the Electron application, native helpers, bundled Lean runtime, and every nested executable; notarize the final distributable with Apple's current notary service; staple and validate the ticket where applicable; verify strict code signatures and Gatekeeper assessment; and install and smoke-test the exact promoted artifact after downloading it with quarantine metadata on a clean supported Mac. Signing credentials must live in protected release secrets, never in the repository or ordinary logs, and publication must promote the already verified artifact rather than rebuild it. The release must include checksums, provenance, supported systems, known limitations, privacy and support links, and certificate-rotation or revocation procedures. Only after these gates pass may the README offer an ordinary learner a download or GitHub publish a Clarifold prerelease.
